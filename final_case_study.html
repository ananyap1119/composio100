How verdicts work

build_now

Official API, authentication, meaningful reads and writes, independent credentials, and a self-serve path are supported.

build_with_customer_credentials

The integration is technically usable, but each customer must provide or authorize credentials.

build_after_app_review

A marketplace, application, or vendor review is required for the intended distribution path.

limited_toolkit

Only a materially restricted capability set is supported.

unknown

At least one verdict-critical condition remains unresolved in official public evidence.

Verdict confidence describes support for the decision. Evidence completeness describes how many requested fields have validated evidence. Secondary gaps such as MCP, WebSocket, or bulk operations do not automatically invalidate a well-supported verdict.

Methodology and validation

The pipeline normalizes identity, retrieves official documentation, extracts field-level claims, validates source IDs and excerpts, applies deterministic verdict rules, then separates machine-final records from review-required records. Calibration reached 8/10 verdict accuracy before production scaling. The final validation run passed 73 tests with clean Ruff checks, and retained-source reuse reduced unnecessary searches.

Limitations

Only publicly available official documentation was accepted. Private partner portals and enterprise documentation may contain additional evidence. Documentation changes over time. Absence of public evidence is not proof that a capability does not exist. Unknown records need targeted vendor or customer confirmation. This is documentation-based buildability, not implementation testing for every app.

Conclusion

The atlas makes connector planning easier by showing where API capability is ready, where customer authorization is the real dependency, and where distribution review is the bottleneck. Unknown records should be routed to focused vendor or customer confirmation rather than treated as failures.
